The user's testing philosophy in this codebase: tests assert that exceptions are thrown when expected, and the **exception message is put under version control** (the `*_Exceptions.txt` diagnostics files). That makes "the expected exception, of the expected type, with the expected message, was actually thrown" a powerful regression check.

**Confirmed 2026-07-14, and broader than exceptions:** the version-controlling of `output/TestSummaries/`, `output/DiagnosticsOutput/`, and the exception files is *deliberately* a *second verification layer outside the test framework itself*. Post-run git diffs reveal regressions the framework cannot report from within — e.g. a capability gate wrongly going false on a 4.3+ platform silently reduces suites to zero checks while staying green, but shows up as a check-count drop in the committed summaries. When assessing "would this failure be caught?", account for this outer layer, not just in-framework checks — and don't propose removing/ignoring the committed outputs as build noise.

**sequoia test taxonomy (explained 2026-07-14):** when creating tests via the runner (`./TestAll create ...`), pick the kind by subject: **`free`** = tests of *free functions* (e.g. `MessagesFreeTest`, `VersionFreeTest`, `CastsFreeTest`); **`regular`** / **`move_only`** = tests of *classes*, with much richer structure — the framework exhaustively checks copy/move semantics (used in some graphics tests via `graphics_move_only_test` etc. in GraphicsTestCore.hpp); further kinds exist (`regular_allocation`, `move_only_allocation`, `performance`). `./TestAll --help` and `./TestAll create --help` enumerate the options. The scaffolding also auto-registers in CMakeLists/TestAllMain and re-runs CMake. Caveat from the user: the tooling works well for many cases but is **work in progress with significant planned improvements — in the sequoia project, not this one** (same boundary as the build-system fixes: don't propose sequoia-tooling improvements here).

**`STATIC_CHECK` rationale (explained 2026-07-14):** sequoia's `STATIC_CHECK(...)` macro (expands to `check("", [&](){ static_assert(__VA_ARGS__); return true; }())`) exists so compile-time checks are *logged into the versioned summary output* — extending the outer deleted-check protection layer to static assertions. The user generally avoids macros and experimented with a non-macro alternative (function template with a `bool` template parameter), but rejected it: the expression collapses to a bare `false` at the call site, so failed-assertion diagnostics (observed on clang) were far worse than raw `static_assert`, whose rich diagnostics need the original expression textually inside the assertion — which only the macro preserves. **Don't propose de-macroing `STATIC_CHECK` or replacing it with raw `static_assert`** — both halves (logging, diagnostics) are load-bearing.

**False-negative-mode semantics (explained 2026-07-14):** in sequoia's `test_mode::false_negative`, *every check must fail for the test to pass* — a passing check fails the FN test (FN tests verify the framework detects failures). Consequence: always-passing infrastructure checks (e.g. characteristics-consistency checks) added to shared test bases must be excluded in FN mode (`if constexpr(Mode != test_mode::false_negative)`). Remember this before adding any check to code paths shared across test modes.

**Output-file specialization is a last resort, not a convenience (stated 2026-07-14):** sequoia supports differently-named output files per platform/build-configuration/driver (the `_Win_Intel_OpenGL_4_6`-style suffixes in the graphics tests). The user *dislikes* exploiting this: wherever possible, tests should be independent of platform, build configuration, etc.; the specialization mechanism exists for when independence genuinely isn't achievable. Don't casually propose "add a config-specific expected file" as a fix — first ask whether the test can be made invariant. Related plan: the user intends to also run the tests in **non-debug GL contexts, likely tied to NDEBUG release builds** — this may change test output (more specialized files) and may require disabling some checks; it also means capability gates legitimately close on *all* platforms in such runs (any gate-consistency check must re-derive from version *and* requested debug mode, not version alone).

But that coverage is **not pursued unconditionally**. Making an implementation-detail function testable sometimes means pulling it out of an anonymous namespace and into a header, which leaks an unwanted dependency. When that trade tips the wrong way, the user deliberately leaves the helper untested rather than damage the dependency structure. This tension is itself lecture content.

**Why:** raised over the lecture-48 removal of `unique_image::to_unsigned` (and the `curlew::validate` / `get_current_resource_index` helpers). They threw `std::logic_error` / `std::runtime_error` on negative input but were untested — *because* they sat in anonymous namespaces and making them testable would have generated unwanted dependencies. Lecture 48's generic utilities (`checked_conversion_to`, in a normal header with its own `ArithmeticCastsFreeTest`) give "the best of both worlds": the negative-value check is now tested *once*, generically, and the call sites just reuse it — so the per-site helpers could be deleted, and the exception type unified to `std::domain_error`, without losing coverage.

**How to apply:** Don't reflexively flag an untested anonymous-namespace helper as a coverage gap, nor suggest hoisting it into a header so it can be tested — the user has likely already weighed testability against dependencies and chosen deliberately. The constructive move is the lecture-48 one: if the same un-tested logic recurs at several sites, factor it into a genuinely generic, independently-tested utility, which earns coverage *without* the dependency cost.
