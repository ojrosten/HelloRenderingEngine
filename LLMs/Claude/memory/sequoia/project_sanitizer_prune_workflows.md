Designed 2026-07-21 (session on `more_claude_convos`); summary in `LLMs/Claude/SEQUOIA_TOOLING.md` §6, actionable rows in `OPEN_ITEMS.md` §1. All of it lands in the sequoia repo (`build_system/`, `DependencyAnalyzer`); avocet inherits via the presets `include`. Organizing principle: **put each axis at the latest pipeline stage it affects** — sanitizers change codegen (configure stage); prune changes only the invocation of an existing binary (build-target stage). Only configure-stage axes multiply build trees.

**Sanitizer axis.** The composable mechanism already half-exists: `base-unix` splices env-var *slots* into `CMAKE_CXX_FLAGS` (`$env{CXX_ASAN_FLAGS} $env{CXX_COVERAGE_FLAGS}...`) and `base-asan-unix` is an environment-only mixin filling one slot. Generalize: add `$env{CXX_UBSAN_FLAGS} $env{CXX_TSAN_FLAGS}` (+ linker twins) to the splice; add mixins `base-ubsan-unix` (`-fsanitize=undefined -fno-sanitize-recover=undefined` — **no-recover is essential**: default report-and-continue is silent in this framework, stderr isn't versioned) and `base-tsan-unix` (`-fsanitize=thread`; exit code already non-zero on reported races → `run` target fails → loud). One slot per sanitizer is load-bearing: preset multiple inheritance unions *disjoint* environment maps, so `inherits: [toolchain, base-asan-unix, base-ubsan-unix]` composes. The combinatorial explosion is illusory: ASan⊥TSan (compiler-enforced), MSan impractical, MSVC has ASan only (Windows column exempt, like coverage), UBSan composes with all ⇒ blessed legs are just **asan+ubsan** and **tsan** (or tsan+ubsan), Linux/Mac. Ad-hoc combos need no preset — `$env{}` falls back to the process environment (`CXX_TSAN_FLAGS=... cmake --preset linux-pkg-gcc`) — but land in the plain preset's build tree (binaryDir keyed by presetName), so env-injection is for one-offs only. Residual cost ≈ 30 mechanical JSON lines per blessed leg. TSan leg is the missing FN-mode evidence channel for the latch/barrier race-amplification harness ([[sequoia-roadmap]] item 6).

**Prune workflow.** Workflows have no test step — tests run via the `run` custom target (`sequoia_set_run_target`, `Utilities.cmake:70-75`), which already threads the `EXEC_ARGS` cache variable. A configure preset setting `EXEC_ARGS=prune` is the wrong move twice over: binaryDir embeds presetName → fresh build tree → full rebuild; and the prune ledger is keyed by build dir (`output/Prune/<relative(buildDir, buildRoot)>/<stem>.prune`, `ProjectPaths.cpp:263-271`) → empty ledger too. Right shape: a **second custom target** `prune` beside `run` (`COMMAND $<TARGET_FILE:${executable}> prune ${EXEC_ARGS}`), plus per-toolchain `base-prune`/`prune-<toolchain>` build presets and workflows reusing the *existing* configure presets. Composes freely with the sanitizer axis (per-tree ledgers can't cross-credit). Cautions: (1) the 2026-07-16 **prune quartet** (silent under-selection) is elevated by blessing a prune workflow — fix before trusting it, same dynamic as `-locate` elevating Tier-1 #4; (2) confirm fresh-tree behaviour degrades to run-everything; (3) `${EXEC_ARGS}` expands unquoted — multiple args must be semicolon-separated, or `separate_arguments()` in the function; (4) `exectuable` typo at `Utilities.cmake:70` (already in the 2026-07-16 typo list).

**Dependency-analyser contract (user TODOs, 2026-07-21).** Two items: (1) **warn when a cpp is not dependent on its associated hpp** — the same-stem association is heuristic, and a cpp that doesn't (even transitively) include its own header signals either a spurious pairing (cf. Tier-1 #2's different-directory collision) or a layout violating the precondition; prune's inferences are unsound either way, so make the implicit contract loud. (2) **Document the precondition itself** (previously nowhere in our records or sequoia's docs): *definitions corresponding to a declaration in an hpp must live either in that hpp or in the cpp of the same stem* — this is what licenses the furnishing step (hpp inherits its cpp's staleness/dependencies, `DependencyAnalyzer.cpp:265-275`). Extension families are already `{.h,.hpp,.hxx} × {.cpp,.cc,.cxx}` (`:77,84`), pairing by identical stem; user notes possibly more options, and a natural generalization when the modules migration lands ([[sequoia-roadmap]] item 2, which forces the prune overhaul anyway).

**Why:** settled design ready to implement offline; avoids re-deriving the axis placement, the EXEC_ARGS trap, and the ledger keying; the precondition is a doc gap that the warning turns into a checked contract.

**How to apply:** when sanitizer/prune preset work or DependencyAnalyzer changes arrive via dependency updates, check them against this design; don't propose configure-level prune presets or shared-slot sanitizer lists; treat the precondition as the contract when reasoning about prune correctness.
