The runner side of `dependencies/sequoia/Source/sequoia/TestFramework` (as of 2026-07). Checking machinery: [[sequoia-checks-semantics]]. Narrative doc: `TestFrameworkPhilosophy.dox` (in-source; `docs/` is just generated Doxygen). Planned changes to much of this: [[sequoia-roadmap]].

**Registration (slated for retirement)**: a hand-written `main` constructs `test_runner{argc, argv, copyright, codeIndent, project_paths::customizer}` then calls `runner.add_test_suite("Name", test_type{"Instance Name"}, ...)` per suite and `runner.execute(timer_resolution{1ms})`. Every test appears ~3├Ś: header `#include`, type name, display-name string (sequoia's own `TestAll/TestMain.cpp` is ~440 lines of this). The `project_paths::customizer` lets clients name their source folder and add `additional_dependency_analysis_paths`. Duplicate detection is runtime. Copyright/indent are ctor params because the runner doubles as the test-*creation* tool (stamped into generated files).

**Key types**: `test_runner` (`TestRunner.hpp/.cpp`) owns `m_Suites` ŌĆö a `maths::directed_tree` of `suite_node{log_summary, optional<test_vessel>}`; results roll up via depth-first traversal. `m_Filter` is an `object::granular_filter` over suite names (`test`/`t`) and source files (`select`/`s`), tracking unmatched selections for warnings. `test_vessel` is manual type erasure ("soul"/"essence" inner classes). Test hierarchy: non-templated `test_base` ŌåÆ `basic_test<Mode, Extender>` (Extender adds check families; "will become variadic once variadic friends are adopted"). The `concrete_test` concept requires `run_tests()`, `source_file()` (usually via `std::source_location` ŌĆö this is the identity used for selection and prune), `reset_results()`. Tests are *re-constructed* during `extract_suite_tree` with full runtime context; the client ctor supplies only the name. Suites are `object::suite` (general-purpose, arbitrarily nestable) ŌĆö but note in-code TODO: only the top-level suite name is stamped on tests (`TestRunner.hpp` ~397).

**Prune / dependency analysis** (`DependencyAnalyzer.hpp/.cpp`; `prune`/`p` option, optional `--cutoff "namespace"`): runs in the ctor, populating the filter with stale tests. Builds a `maths::directed_graph` of file dependencies by scanning `Source/`, `Tests/`, and additional paths for hard-wired extensions (`.cpp/.cc/.cxx`, `.hpp/.h/.hxx`) and lexing `#include` directives with a hand-rolled comment-aware lexer (in-code `// TO DO: Bug here with #endif` ~line 150). Same-stem heuristics pair `Foo.cpp`Ōåö`Foo.hpp`; staleness (mod-time vs prune stamp) propagates transitively; **test materials count as dependencies**; a "your executable is out of date; please build it!" tripwire fires if any file is newer than the test binary. Set to run = stale Ōł¬ previously-failing. Persisted state under `output/Prune/`, stem-discriminated per build dir: `.prune` stamp, failures file, selected-passes file (so filtered runs don't re-run passing selections forever), `.external` unresolved-includes list. Entirely `#include`-based ŌĆö cannot see `import`; will need an overhaul for modules ([[sequoia-roadmap]]).

**Test/project creation CLI** (`TestCreator.*`, `ProjectCreator.*`, `FileEditors.*`): `create`/`c` with six species ŌĆö `regular_test`, `move_only_test`, `regular_allocation_test`, `move_only_allocation_test`, `free_test`, `performance_test`; sub-options `--suite`, `--header`, `--test-class-forename`, `gen-source/g`, `--framework-diagnostics`; chainable, then reruns CMake. Nascent tests dispatch through `nascent_test_factory = object::factory<...>` + `std::visit` ŌĆö the factory pattern intended for future registration is already deployed here. Templates in `aux_files/` (`TestTemplates/`, `SourceTemplates/`, `ProjectTemplate/`) with `?forename`/`?Class`-style textual substitution. Generated tests are wired in by **textual pattern-matching edits** of the common-includes header, the main cpp (`add_to_suite` matches the `add_test_suite(...)` call), and CMakeLists (`add_to_cmake`) ŌĆö breaks if those files are reformatted. `init` scaffolds a whole project from `ProjectTemplate` (optionally first build, git init, IDE launch).

**CLI parsing** (`Source/sequoia/Parsing/CommandLineArguments.*`): home-grown and tree-structured ŌĆö an options forest (`maths::directed_tree` of `option{name, aliases, parameters, early/late executors}`) parsed into an operations forest, invoked depth-first (`late` runs after children ŌĆö how `create`'s sub-options finalize a nascent test). Help text auto-generated. The runner's ~250-line nested-lambda option table in `process_args` is dense ŌĆö a known "older code" locus.

**Execution**: concurrency modes `dynamic` (default; `std::execution::par`, hand-rolled thread-pool fallback on clang/libc++), `--serial`, `--thread-pool N`. Non-parallelizable tests (opt-out trait; performance tests always) are sorted to the front and run serially ("TO DO: replace with stable_sort", `TestRunner.cpp` ~808). `recover`/`dump` force serial and write crash-forensics breadcrumbs to `output/Recovery/`. Instability analysis: `locate-instabilities N [--sandbox]` runs everything N times (sandbox mode re-invokes the executable itself via `runtime::shell_command`), aggregating prune files (failures = union, passes = intersection).

**Output layout**: `output/DiagnosticsOutput` ŌĆö version-controlled FP/FN and caught-exception text, with optional per-test `output_discriminator()`/`summary_discriminator()` hooks for platform/GPU discrimination. `TestMaterials/<mirror of test source path>/` with `WorkingCopy`/`Prediction`/`Auxiliary`; working copies land in `output/TestsTemporaryData`; `update-materials`/`u` soft-updates prediction from working on failure. Per-test summaries under `output/TestSummaries`.

**Strengths to remember**: the framework dogfoods sequoia's graph/tree/traversal machinery for suites, CLI parsing, and dependency analysis; FP/FN self-testing; per-build-dir, materials-aware prune state.
