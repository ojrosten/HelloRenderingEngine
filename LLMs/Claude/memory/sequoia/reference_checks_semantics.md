The checking layer of `dependencies/sequoia/Source/sequoia/TestFramework` (as of 2026-07). See [[sequoia-test-framework-core]] for the runner and [[sequoia-overview]] for orientation.

**Check flavours are tag types** dispatching overloads of free `check(...)` (`FreeCheckers.hpp`): `equality`, `simple_equality` (`==` only, no tester recursion), `equivalence`, `weak_equivalence`, `with_best_available`. The fallback chain is encoded structurally — `weak_equivalence_check_t::fallback = equivalence_check_t`, whose fallback is `equality_check_t`; the generalized overload walks it when no direct tester overload exists.

**Semantics of the flavours**: equality = same type T (and if `T` is deep-equality-comparable, *both* `operator==` and any `value_tester` equality test run regardless of each other's result — deliberate fidelity, either could be buggy). Equivalence = check T against a *different* type U capturing T's full logical state (e.g. `fs::path` ≡ same name + file contents). Weak equivalence = U captures a proper subset (e.g. `std::function` nullness). `with_best_available` picks the strongest supported check per type via static reflection — key for heterogeneous aggregates.

**`value_tester<T>`** is the central client customization point: specialize with static `test(equality_check_t, test_logger<Mode>&, obtained, prediction)` (and/or equivalence overloads with U ≠ T; optional `tutor` advisor arg). Recursion is natural — testers call `check(equality, "field", logger, ...)` per field. If no tester exists but the type is a `faithful_range`, an iterator-range overload runs. `general_equivalence_check_t<Customizer>` carries a value-based customizer through to the tester (e.g. per-extension file comparers for path equivalence). Stock testers in `ConcreteTypeCheckers.hpp` cover string/pair/tuple/path/variant/optional/any/smart pointers/`std::function`/`chrono::time_point`.

**`check_semantics`** (regular: `RegularCheckers.hpp`; move-only: `MoveOnlyCheckers.hpp`): given `x != y` (prerequisites are *checked*, not assumed), verifies ordering-operator coherence (incl. `<=>`), copy/move ctor, copy/move assign incl. self-assign, swap incl. self-swap, and — activated purely by static reflection when T is (de)serializable — serialization round-trip coherence. Optional `Mutator` catches shallow copies; optional moved-from-state values. Move-only variants take `T&&`s plus `xEquivalent`/`yEquivalent` clones or maker function objects. Every step is a `check_foo`/`do_check_foo` pair taking an `Actions` object so allocating and non-allocating types share one skeleton (`SemanticsCheckersDetails.hpp`).

**Allocation testing**: `shared_counting_allocator<T, PropCopy, PropMove, PropSwap>`; `allocation_info` bundles an `alloc_getter` + `alloc_prediction<Event>`s written with UDLs (`1_c`, `0_mu`, `1_pc`, `1_pm`, `1_anp`, `1_ma`...; "para" = allocator-extended ctors). `alloc_prediction_shifter` + `allocation_equivalence_classes` keep predictions platform-independent (MSVC debug builds allocate extra). `regular_allocation_test`s define `template<bool PropCopy, bool PropMove, bool PropSwap> void test_allocation()`; `do_allocation_tests()` runs all 8 propagation combinations (4 for move-only).

**Test modes** (`TestMode.hpp`, NTTP threading through logger/sentinel/checker/test): `standard`, `false_positive` (checks that should pass — tests of the framework/testers), `false_negative` (pass/fail *inverted at top level* — the test passes when the underlying check fails, proving a tester actually detects breakage). Both diagnostic modes dump their output to version-controlled files so the failure *messages* are themselves regression-checked. Idiom: every client `value_tester` deserves FN-mode diagnostics.

**Test base classes** (all × standard/false_positive/false_negative): `free_test`, `regular_test`, `move_only_test`, `regular_allocation_test`, `move_only_allocation_test`, `performance_test` — all `basic_test<Mode, Extender>` (`FreeTestCore.hpp`, which also owns `working_materials()`/`predictive_materials()`/`auxiliary_materials()`). Extenders bolt member `check_semantics`/etc. onto `checker<Mode, Extender>` via deducing-this — CRTP-free static dispatch. `STATIC_CHECK` is a macro inside `checker`: a `static_assert` that also registers as a runtime-counted check.

**Other checks**: `check_exception_thrown<E>` passes only on exactly E, logs the caught message to a version-controlled file (postprocessor strips project root for platform independence). Relational: `check(Compare, ...)` accepts any binary predicate with a `failure_reporter<Compare>` specialization for messages; `within_tolerance` supplied; comparators lift over ranges. Performance: `check_relative_performance` — randomized paired trials, trimmed mean ± sample sd, retry with growing trials; performance tests are marked non-parallelizable. State-transition testing: `transition_checker<T>` (`StateTransitionUtilities.hpp`) walks a `maths::directed_graph` of object states (nodes = generators, edges = transitions) — cures the "linear unit test" problem.

**Output**: `test_logger<Mode>` is mutated solely through RAII `sentinel<Mode>` (nested sentinels → indented hierarchical messages with per-level type info via `type_demangler`; escaping exceptions become critical failures with context). `tutor<Advisor>` appends "Advice:" on failure and propagates through nested tester calls. Optional recovery files leave crash-forensics breadcrumbs.

**Cross-cutting idioms**: tag types + concept-constrained overload sets rather than enum switches; capability detection over configuration (reflection-activated serialization/ordering checks); version-controlled output files as an outer regression layer.
