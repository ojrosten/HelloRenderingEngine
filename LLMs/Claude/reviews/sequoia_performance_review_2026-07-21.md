# Sequoia: performance-testing support — heuristic assessment & hypothesis-testing replacement

**Date:** 2026-07-21
**Scope:** `Source/sequoia/TestFramework/PerformanceTestCore.hpp/.cpp`, `Source/sequoia/Maths/Statistics/StatisticalAlgorithms.hpp`, `Tests/TestFramework/PerformanceTestDiagnostics.cpp/.hpp`, plus perf-machinery customers found by grep.
**Method:** two parallel agents — a coder (mechanics/integration/coupling) and a mathematician (statistical methodology and replacement evaluation) — with the orchestrating session verifying the predicate, retry loop, pairing-destruction, and P5 claims directly against source. Prior-review overlaps (2026-07-16 Tier-3 perf item; assorted nits) attributed, not re-raised.
**Status/register:** exploratory — the user self-describes the acceptance criterion as a physics-inspired heuristic ("when should two distributions be considered separate"), "fairly crude (albeit reasonably effective in practice)", and is **considering replacing it with a hypothesis-testing-based approach**. This review is input to that design decision, not a defect list against settled code.

**Verdict up front.** The heuristic's *logical shape* is right and each ingredient has a named formal counterpart (interval-exclusion non-inferiority test; randomized paired design; trimmed estimators; conjunctive group-sequential retry). Exactly one quantity is uncalibrated: the band uses the SD of individual timings instead of the standard error of a paired statistic — making the test statistically inconsistent (more trials buy nothing) and powerless against marginal regressions, while remaining a near-deterministic gross-regression tripwire (which is where "reasonably effective in practice" lives). The mathematician's bottom line: **"'Underwhelmed' is not the verdict; 'one plug-in quantity short of a real test' is."** Two warnings for the replacement: textbook TOST would *invert the burden of proof* (fail-unless-proven-equivalent — chronic false failures under CI noise; the current pass-unless-excluded burden is correct and must be preserved), and an SE-based band is only honest if the data are **paired** — unpaired SE is dishonest under autocorrelation/drift, which the current SD band is accidentally robust to. The coder's coupling map shows the swap is well contained: one function body, its doc, the summary-text format plus its postprocess/pinning shadow, and the diagnostics' expected outcomes; the scaffolding (timing, logging, three-layer mode inversion, creation CLI) is genuinely predicate-agnostic.

## Part I — The procedure as mathematics (verified against `PerformanceTestCore.hpp:77-196`)

Parameters: predicted speed-up interval `[r⁻, r⁺]`, `1 < r⁻ ≤ r⁺`; `trials = n ≥ 5` (default 5); `num_sds = k > 1` (default 4); attempts `A` (hard-coded 3 in `performance_extender`, hpp:245).

Attempt `a` (fresh data each attempt; `N = n·a`, hpp:110): each trial runs both tasks back-to-back on the calling thread in coin-flipped order (`std::random_device`, hpp:116-131); wall-clock (`steady_clock`) durations recorded into separate vectors, which are then **sorted** (hpp:141-142 — destroying the pair correspondence), trimmed by exactly one min and one max, and summarized by trimmed mean `m` and trimmed SD `σ` (Gaussian-bias-corrected `N−1.5` divisor, `StatisticalAlgorithms.hpp:109-131`).

Acceptance (hpp:147-163): **gate** `m_f + σ_f < m_s − σ_s` (1σ per side, independent of `k`), then a band check built from the **smaller** of the two SDs: if `σ_f ≥ σ_s`, pass ⟺ `[r⁻·m_f, r⁺·m_f] ∩ [m_s ± k·σ_s] ≠ ∅`; else pass ⟺ `[m_s/r⁺, m_s/r⁻] ∩ [m_f ± k·σ_f] ≠ ∅`. Retry (hpp:186-191): break on first pass (standard mode) / first fail (FN mode — correct mode symmetry); overall pass = any attempt passes.

Physics-instinct → formal-counterpart map: non-overlapping error bars → two-sample separation test (uncalibrated); k·σ band → significance via Gaussian tails (miscalibrated: σ of individuals); **two-sided band → equivalence testing, with the upper bound doubling as a dead-code-elimination tripwire (one of the design's best features — preserve it)**; min/max trim → trimmed estimators; coin-flipped order → randomized design; `N−1.5` → Cochran-type bias correction; retry loop → conjunctive group-sequential design (independent stages, escalating N, false-failure probabilities multiply — the right CI economics, built by instinct).

## Part II — Statistical findings (PS-series)

- **PS1 [builds on 2026-07-16 Tier-3, quantified] The band doesn't narrow with N ⇒ the test is statistically inconsistent.** `k·σ` of individual timings (k=4) vs sampling error of the mean `σ/√(N−2)` (≈0.35σ at N=10): band false-failures are near-impossible, but a shortfall smaller than ~4σ of single-run noise is undetectable *forever* — escalation n→2n→3n buys nothing. A candidate delivering 1.6× against claimed [2,3] can pass indefinitely on a noisy machine.
- **PS2 The gate is sharpest exactly where fine discrimination isn't needed.** Near-deterministic against gross regressions (the practical effectiveness); for r⁻ close to 1, single-run noise defeats it even when the claim is true, and more trials never rescue it. Effective hidden minimum detectable ratio ≈ `1 + (σ_f+σ_s)/m` per attempt.
- **PS3 Retry-until-pass = conjunctive sequential test with uncontrolled per-stage levels.** Structure is *good* (independent stages, α's multiply); the costs: per-stage α unknown, and power multiplies too — per-stage power p ⇒ overall detection pᴬ (0.8 ⇒ 51% over 3 attempts). Type I crushed, Type II compounds silently ("both levers push toward passing", 2026-07-16, made precise). Escalating N would normally offset this; PS1 means it doesn't.
- **PS4 Fixed-count trim decays as N grows:** 20% per side at N=5 → ~7% at N=15, so later (larger) attempts are *more* contaminated by e.g. scheduling stalls. Fraction-based trim (~10%/side) is the fix; the trimming instinct itself is right.
- **PS5 [verified] The pairing is collected and then destroyed.** Each trial measures both tasks adjacently (common-mode drift shared within the pair); the sort at hpp:141-142 severs the correspondence and the analysis compares unpaired means, so all common-mode variation lands in σ_f, σ_s. Per-trial ratios `d_i = ln(s_i/f_i)` would cancel it — **the single largest free lunch in the design**, and the precondition for an honest SE-based band.
- **PS6 Predicate structural oddities** (would become visibly wrong if the band were calibrated): band always from the *smaller* SD with the projected mean's scaled uncertainty dropped (proper propagation: `√(r²σ_f² + σ_s²)`); branch-on-larger-σ asymmetry unprincipled; `num_sds` validated `> 1` with no upper sanity bound (hpp:86-87).

**Net characterization:** a gross-regression tripwire with a wide uncalibrated equivalence band. Type I ≈ 0 for r⁻ ≳ 1.5; power ≈ 1 for catastrophic regressions, ≈ 0 for marginal ones. The user's self-assessment is, on the numbers, correct: the crudeness costs power and consistency, not validity of what it catches.

## Part III — Replacement evaluation & recommendation

**Framing decision that dominates:** keep the burden of proof as-is — fail iff a confidence interval for the true ratio is *disjoint* from `[r⁻, r⁺]`. Textbook TOST inverts this and would degrade the framework. "More rigorous" must mean *calibrating the interval*.

Candidates: **Welch's t (unpaired)** — rejected: ignores free pairing, SE dishonest under autocorrelation/drift, slow normality at N≤15. **Mann-Whitney/rank** — rejected: tests dominance not a ratio; margin encoding awkward; uninterpretable failure text. **Bootstrap/permutation** — rejected at these N (shaky coverage below ~15-20 atoms, RNG nondeterminism in pass/fail, most code); revisit only if the parametric route proves inadequate. **Min/low-quantile comparison** — theory noted (min concentrates fast under additive contamination — the `timeit` folklore) but fragile under frequency scaling (minima may come from different frequency states); low quantiles keep robustness, though pairing already removes the drift they'd defend against. **Full SPRT** — over-engineered; but **group-sequential conjunctive design is what the code already does** — the principled version is the current loop with a calibrated per-stage test. **Sign test on paired ratios** (`#{d_i ≥ ln r⁻}`, exact binomial) — best distribution-free fallback/cross-check; most human failure message.

**Recommendation (~30 lines against the existing statistics module):**
1. **Pair:** `d_i = ln(slow_i/fast_i)` per trial (keep randomized within-pair order — it's what validates the pairing under trends).
2. **Calibrate:** pass ⟺ `[d̄ ± c·s_d/√N]` intersects `[ln r⁻, ln r⁺]`, with ~10%/side fractional trim on the `d_i`. Subsumes the gate and both band branches; CI below interval → "too slow"; above → keep the "suspiciously fast — check for elision" message. The band now narrows as √N; escalation finally works.
3. **Keep the retry loop; document it as what it is** — a 3-stage conjunctive group-sequential design. c≈3 per stage ⇒ nominal overall false-failure ~3×10⁻⁸ (model-error-dominated, CI-safe) with per-stage power *growing* by stage. Document overall detection = ∏ per-stage power; consider a 2-of-3 rule (Type I ≈ 3α², power p²(3−2p)) if marginal-regression detection starts to matter.

Defaults: `trials = 10`, `c = 3`, `maxAttempts = 3`, trim 10%/side. Failure message in user units: *"measured speed-up (geometric mean): 1.34×, 99% interval [1.22, 1.47]; predicted [2.0, 3.0]; N = 15 paired trials, attempt 3 of 3"* — replacing per-task `mean ± k·σ` (single-run scatter nobody can act on).

## Part IV — Coder findings (P-series)

- **P1 [medium] A throwing task aborts the whole test with no check registered.** Measurement loop (hpp:108-132) precedes sentinel construction (hpp:194) [verified]; a throw from `fast`/`slow` escapes with no `log_performance_check`, gets generic critical-failure attribution, and kills the remaining checks in the test. Regular checks attribute-and-continue. Fix shape: try/catch around the loop constructing a sentinel from `description` alone. Same family as A2 (transition_checker) — a house pattern worth fixing once.
- **P2 [medium] Under instability analysis, a stably-regressed perf check reads as flaky.** Failure messages embed timing text; `analyse_output` compares messages by equality (`FailureInfo.cpp:51`), so N reps of a consistent regression produce an N-way "Instability detected" split. The `postprocess` smoothing exists only on the versioned-diagnostics path. Any replacement inherits this unless failure text is stabilized or the comparison taught the same forgiveness. Known/accepted?
- **P3 [medium-low] No result-observation facility for measured callables.** `profile` discards results; no `do_not_optimize` affordance. Traced: full elision ⇒ gate fails loud; fast-side-only elision ⇒ band fails loud — **no silent false-pass path**, but a CPU-bound benchmark gets a baffling failure, not a diagnosis. All in-tree customers measure sleeps/thread-pool work; the machinery is unvalidated against pure computation. Decide before a statistics upgrade implies benchmark-grade: documented contract ("tasks must have observable side effects") or an `observe(...)` helper.
- **P4 [low] Default `trials = 5` runs the statistics on 3 points** after trim — and 5 is also the enforced minimum (hpp:92-93). Any t-based band would force this default up anyway (recommendation says 10).
- **P5 [low; verified] `calibrate` at namespace-scope static init + truncating `inverse`.** `const static auto delta_t{calibrate(ms{5})}` (`PerformanceTestDiagnostics.cpp:16`) pays ~35-70 ms of timed sleeps at binary launch even when perf tests are filtered/pruned away — function-local static confines it. `constexpr auto inverse{Period::den / Period::num}` (hpp:223) is integer division: zero for any `num > den` period (seconds, minutes) ⇒ zero target from a public template. Only milliseconds used in-tree.
- **P6 [low] `acceptable_mismatch`'s `rfind` window is not line-anchored** (cpp:26): a non-timing mismatch is forgivable if a `"Task duration:"` line precedes it in the window. Currently shielded *accidentally* (timing lines always differ, resetting the window); fails if a timing line compares byte-identical or a description contains the phrase. Cheap hardening: require the `rfind` hit to fall after `pos`'s line start. No postprocess test pins this corner.
- **P7 [low] Contract gaps:** `maxAttempts` hard-coded to 3 in the extender and absent from its signature (the one lever a noisy-CI user wants); the per-mode retry polarity means FN-mode tests must contain only designed-to-fail perf checks (undocumented); doc says `maxSpeedUp > minSpeedUp` but code permits equality and the FN diagnostics *rely* on `(2.0, 2.0)` (doc wrong); the growing-trials schedule (`k·trials` fresh per attempt) documented nowhere.

## Part VI — Diagnostics notes

FN cases (equal tasks ×2; ratio-4 against (2.0, 2.5) exercising the upper bound) are low-flake by construction and the FN-polarity retry absorbs spurious passes. FP cases (true 2× against (1.8, 2.1); true 4× against (3.4, 4.1)) are the genuinely stochastic ones; sleep overshoot pushes the observed ratio toward the 1.8 edge — works, but thinner margin than it looks. `postprocess` is pinned by 13 deterministic string cases (P6 corner not among them). `TestRunnerPerformanceTest.cpp` doesn't use this machinery at all (extracts "Execution Time", `within_tolerance`) — the runner's own perf pinning gains nothing from the swap.
