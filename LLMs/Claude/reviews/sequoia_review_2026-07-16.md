# Sequoia: strengths, weaknesses, gaps

**Date:** 2026-07-16
**Reviewed state:** `dependencies/sequoia` working copy (repo tip dated 2026-03-11)
**Method:** six parallel Claude review agents (runner infrastructure; checking layer; Core/Object + Meta + small modules; DataStructures + ContainerUtilities + Concurrency; Maths; test-coverage/meta-quality), each reading the actual sources with findings labelled CONFIRMED (traced end-to-end, several verified by compiled repros under g++ 15/WSL with ASan/UBSan) or PLAUSIBLE. Top claims spot-checked independently before synthesis. Calibration applied: defensive throws are intentional tripwires; no speculative compiler-warning concerns; Tests/ treated as real customers; findings in registration/prune/creation-CLI territory tagged **[roadmap-covered]** (planned rework: reflection-based registration via `object::factory`, modules migration forcing a DependencyAnalyzer rewrite, test-creation CLI refresh); Maths/Spaces and Physics assessed at intent level only (acknowledged flux).

**Verdict up front.** The architecture is genuinely strong — several ideas (the equivalence taxonomy, FN/FP mode inversion, the Actions pattern, per-build-dir materials-aware prune) are better than the state of the art, and "very well tested" is an earned claim *for the test artifacts*. The findings cluster in three places: (1) a handful of real, confirmed bugs — one of which (graph copy construction) is only masked by an optional compiler optimization; (2) the prune subsystem, where three confirmed bugs all push in its worst failure direction, silently skipping or mis-crediting tests; (3) the verification *process* — no CI, no TSan — which is the single highest-leverage improvement available and costs no new tests at all.

## Strengths (verified, not taken on trust)

- **The equality/equivalence/weak-equivalence taxonomy** with its structurally-encoded fallback chain, and the deliberate decision to run `operator==` *and* the `value_tester` independently (either could be buggy). The fallback direction is toward stronger relations — the safe direction.
- **The FN/FP self-testing scheme** is real, not token: ~30 FN diagnostic files including deliberately broken types (broken semantics, inefficient operations), 295 version-controlled diagnostic outputs, and an end-to-end test that `init`s a project, compiles it with a real toolchain, and drives it through ~15 option combinations including prune, sandbox instability analysis, and materials update.
- **Prune's bookkeeping design**: per-build-dir state, the conservative process-start timestamp, filtered runs updating passes/failures without advancing the global stamp, the executable-out-of-date tripwire. The *design* is more careful than most build systems'; the bugs below are in the execution.
- **Connectivity's initializer validation and comp-index bookkeeping**: `modify_comp_indices`' stale-vs-corrected disambiguation was traced through loop erasures and sequential erases — the notoriously hard part holds up. Error messages carry coordinates and are themselves regression-tested.
- **`task_queue`'s condition-variable discipline is correct** (no lost/spurious-wakeup hazard), the try-lock stealing paths can neither lose nor double-execute tasks, and shutdown drains rather than drops. `assignment_helper`'s POCCA/POCMA/POCS matrix is right in every branch, including the clever reset-then-swap for POCCA-without-POCS.
- **Statistics avoids catastrophic cancellation** (two-pass variance); concurrent traversal results are deterministic (futures stored in submission order); allocation-prediction UDLs are consteval, per-event type-safe, and platform-normalized.
- Verified sound under targeted suspicion: the sentinel mode-inversion truth table, `post_serialization_action` placement, swap/self-swap prerequisites, `swap_partitions` rotate arithmetic, static_queue circular-buffer arithmetic, factory invariant enforcement, `granular_filter`'s no-short-circuit matched-tracking.

## Correctness findings — tier 1 (confirmed, real-world consequences)

1. **Graph copy construction is only correct under NRVO, which is optional.** `connectivity_base`'s copy ctor (`Connectivity.hpp:212`) initializes `m_Edges{copy_edges(other)}`, and `copy_edges` calls `this->reserve_nodes(...)` (line 1348) and `this->cbegin_edges(...)` (line 1334) — members of an object whose lifetime hasn't begun. It works today because NRVO aliases the local `storage` onto `m_Edges`; **confirmed by instrumented build**, and with `-fno-elide-constructors` a shared-weight `embedded_graph` copy *hangs*. gcc/clang elide even at -O0; whether MSVC `/Od` does is worth checking — if not, debug-build copies of shared-weight graphs are broken on Windows today. Fix is mechanical: operate on `storage`, not `this`, at both sites.
2. **Prune can silently skip affected tests** (`DependencyAnalyzer.cpp:280-287`): header "furnishing" looks only at the lexically-next node, so same-named cpps in different directories displace the pairing and `Foo.hpp` never inherits `Foo.cpp`'s dependencies. Touch a transitive dependency → dependent test not marked stale. The project's own test materials show same-named cpps are a normal state of affairs. [Roadmap-covered by the modules rewrite, but a today-bug for prune users.]
3. **`prune_record` round-trip breaks on paths with spaces** (`DependencyAnalyzer.hpp:34-49`): written unquoted, read via `fs::path`'s quoted single-token extraction — one spaced path corrupts that record *and drops every subsequent record* from `.failures`/`.passes`. Also the tick count is written signed (negative on libstdc++'s `file_clock`) and read as `std::size_t`.
4. **Sandbox-mode pass aggregation is always empty** (`DependencyAnalyzer.cpp:507-529`): the intersection uses `prune_record`'s defaulted `<=>`, which includes `time_stamp` — but each sandbox child stamps records with its own process start time, so identical tests never compare equal. Previously-failing tests that now pass stably under `locate N` stay in `.failures` indefinitely. The end-to-end test uses a deliberately unstable test, so this never surfaced. Fix: intersect with the `path_projector` used everywhere else.
5. **Past-the-end dereference in materials soft-update** (`MaterialsUpdater.cpp:146`): with working/prediction diffs shaped like a rename (every added file sorting before some deleted one), the loop exhausts `in1` and then dereferences it. Any `update-materials` run can hit this; UB in release, iterator assert on MSVC debug. Fix: an `iters.in1 != fromEnd` guard.
6. **Static graph index-type selection rejects valid graphs** (`StaticGraphDetails.hpp:25-32`): the index type is chosen from `Order` (or `Order`/`Size`) but also parametrizes partition offsets that run to `Size` (directed) or `2×Size` (undirected/embedded). **Confirmed**: `static_directed_graph<300, 3>` selects `unsigned char` and every construction throws "Partition index out of bounds". Loud, never silent — but it fires on legal input, and no test constructs near the boundary. Selection should be based on `num_static_edges(flavour, Size)` and comp-index reach.
7. **`pseudo_depth_first` misclassifies loop edges on embedded graphs** (`GraphTraversalDetails.hpp:304-315`): `complementary_index() < dist` compares a forward position against a *reverse*-iterator distance. **Confirmed by repro**: one loop gives `edgeFirst/edgeSecond` counts of 2/0 (BFS correctly gives 1/1). No traversal test contains a loop — the exact hole it escaped through.
8. **Exceptions during output in parallel test runs terminate the process** (`TestRunner.cpp:69-80, 846`): `essence::execute` catches exceptions from `run_tests()`, but the versioned-write/summarize calls sit outside the try/catch, inside `std::for_each(execution::par, ...)` — disk-full in one test → `std::terminate`, no grand totals, no prune update. The clang thread-pool fallback propagates cleanly, so behavior differs by platform. Cheap fix: catch-and-stash inside the executor.

## Tier 2 (confirmed, latent or edge-condition)

9. **`sentinel_base`/`sentinel` have defaulted moves over a raw logger pointer** (`TestLogger.hpp:234-235`): a moved-from sentinel's destructor runs the full bookkeeping again — double `decrement_depth`, potential `logic_error` from a destructor (→ terminate), or silent depth mispairing. Latent (nothing in-tree moves one), but the type is public and `[[nodiscard]]` invites `auto s = std::move(...)`. Delete the moves or null-and-guard.
10. **`checker::checks()`/`failures()` cannot compile** (`FreeCheckers.hpp:842-846`) — they call logger members that don't exist; never instantiated, so it builds until a client touches them. Should be `results().deep_checks`/`.failures`, or deleted.
11. **Serializable move-only types with heterogeneous equivalents hit a deduction dead-end in `check_semantics`** (`MoveOnlyCheckersDetails.hpp:51-54`): the serialization guard checks only that `T` is serializable, not `U == T`; a supported-looking combination is a hard instantiation error.
12. **Contiguous storage returns the whole sequence for out-of-range `partition(i)`** (`PartitionedData.hpp:953-982`) where bucketed returns empty/throws — the two interchangeable storage policies disagree, and the divergent one is unpinned by tests. Relatedly, **`end_partition` et al. are `noexcept` but call a deliberately-throwing tripwire** (lines 530-572) — the diagnostic becomes a message-less terminate exactly when it fires. (`get_begin_iterator` and `get_end_iterator` themselves have the right noexcept split; only the public wrappers are inconsistent.)
13. **`thread_pool` lifecycle edges** (`ConcurrencyModels.hpp`): double-`join()` throws `system_error` (lines 269-273; one-line fix — check `joined` first); `numThreads == 0` → modulo-by-zero UB in `push` (lines 236-248) — a defensive ctor throw would fit house style; partial thread-creation failure in `make_pool` terminates (lines 284-332; classic pre-`jthread` hazard). A `jthread`/`stop_token` migration subsumes all three.
14. **`has_sensible_semantics` is provably inert** (`Iterator.hpp:215-220`): it passes the reference type as `indirectly_writable`'s `Out` parameter, so both conjuncts are false for essentially every instantiation and the guard can never reject the exact misuse its doc comment forbids. Real proxy-policy customers exist (`Connectivity.hpp:1655`, `DynamicTree.hpp:341`, `Factory.hpp:88`).
15. **`static_queue`/`static_stack` declare `X(X&)`/`operator=(X&)` where `X(X&&)` was meant** (`StaticQueue.hpp:37,41`, `StaticStack.hpp:36,40`) — moves silently degrade to copies; `static_priority_queue` has the correct spelling. Also `pop()` on empty stack/priority-queue wraps indices (OOB) while `static_queue::pop` is a checked no-op — three siblings, three contracts.
16. **`to_array` off-by-one** (`ArrayUtilities.hpp:37`): `i > l.size()` should be `>=` — undersized init-lists of non-default-constructible types read one past the end. **`static_linearly_partitioned_sequence` never validates per-partition sizes** (`StaticLinearlyPartitionedSequence.hpp:25` + `PartitionedData.hpp:1132-1159`) — right total, wrong shape reads past an inner initializer list.
17. **`monotonic_sequence`'s `noexcept_spec` constraint is inverted** (`MonotonicSequenceDetails.hpp:25-39`): the specialization that should apply to allocator-aware containers unconditionally returns `true_type`, making the POCS logic dead and `swap` `noexcept(true)` even where the underlying vector swap could throw; for non-allocator containers the primary is a hard error if ever instantiated. The `requires` clause looks like it was meant to be negated.
18. **Parser trio** [roadmap-covered, but they'll shape the CLI refresh] (`CommandLineArguments.cpp`): the end-of-input completeness check ignores `saturated_args`, so a missing nested argument is silently accepted (lines 164-166); concatenated aliases never require a leading `-` and silently ignore the first character (`"qv"` acts as `-v`; lines 224-227); partial concatenated matches commit operations before validation (lines 227-241), and concatenated aliases for options *with* parameters can never collect arguments.
19. **TextProcessing** (feeds test creation): `replace_all` with empty `from` is an infinite loop with unbounded memory (`Substitutions.cpp:120-149`; `replace_all_recursive` also never terminates when `to` contains `from`); `to_camel_case` mishandles consecutive underscores (`Substitutions.cpp:43`); raw-`char` `<cctype>` calls are UB on non-ASCII (`Substitutions.hpp:36`, `Substitutions.cpp:26,38`, `ShellCommands.cpp:25` — the same function does the `unsigned char` cast correctly elsewhere, so clearly accidental); the predicate `replace_all`'s `+1` advances skip adjacent candidates (`Substitutions.hpp:136,140` — intentional boundary reasoning or off-by-one?).
20. **`meta::stable_sort` is unstable for the two-element equivalent case** (`TypeAlgorithms.hpp:252-256`): the singleton `merge` specialization swaps unconditionally when `Compare<T,U>` is false, including equivalence; the general path is stable. Latent for `type_comparator` until two types share a `type_name` — and Physics unit canonicalization sits on top of this. Fix: the specialization should swap only when `Compare<U,T>` holds.
21. **Shell commands never quote paths** (`TestRunner.cpp:772-774`, `ShellCommands.cpp:29,59-62`): sandbox instability analysis breaks under any directory with a space (realistic on Windows); `cd` without `/d` won't change drives; unquoted interpolation into `std::system` is an injection surface. Also `if(cmd.m_Command.data())` (`ShellCommands.cpp:55`) is always-true — `!empty()` was surely meant.
22. **`add_include` destroys non-include content interleaved in an include block** (`FileEditors.cpp:24-77`) — conditional includes, comments between groups; boundaries come from raw `find`/`rfind` of `"#include"`, so the literal in a banner comment shifts the region. `add_to_suite`'s anchoring (line 102) is an unanchored search satisfiable by any later occurrence, and `");"` inside a string argument truncates its region. [Roadmap-covered; latent today because targets are generated files — but invoked on user-editable files every `create`.]
23. **Assorted confirmed smaller items**: `serial::push`'s convertibility constraint is reversed relative to `thread_pool::push` (`ConcurrencyModels.hpp:155`); statistics functions declare `input_iterator` but are multi-pass (`StatisticalAlgorithms.hpp`); the orderable-prerequisite diagnostic has a dead branch and always prints "y > x" (`SemanticsCheckersDetails.hpp:372-377` — fixing churns versioned FN outputs, which exist for exactly that); visible message typos baked into versioned outputs ("Inonsistent para-move", "xand x Equivalent" — `MoveOnlyAllocationCheckersDetails.hpp:47,59`); `cmake_cmd`'s unused `args` branch emits a malformed command (`Commands.cpp:35-38`); diagnostics/instability output paths can collide between tests sharing (suite, source-filename) with no detection (`IndividualTestPaths.cpp:43-55`) — and colliding counts stay divisible by `numReps`, so the instability tripwire doesn't fire; tests sharing a source file across separate `add_test_suite` calls share one WorkingCopy directory with no concurrency guard (`TestRunner.cpp:275-304`); the suite-object `add_test_suite` overload skips `check_for_duplicates` (`TestRunner.hpp:279-286`); `insert_node` rollback uses the unclamped position (`GraphPrimitive.hpp:469-473` — UB in a bad_alloc rollback path); undirected non-shared weight mutation isn't exception-safe and can leave partner halves disagreeing, after which weight-equality partner lookup misbehaves (`Connectivity.hpp:94-123, 267-308`); `insert_join`'s rollback never decrements the comp indices it incremented (`Connectivity.hpp:126-149, 1538-1546` — the in-code strong-guarantee TODO at :763 suggests known-unfinished); the iterator-range check admits single-pass input iterators it would consume-then-reread (`FreeCheckers.hpp:368-416`); `tests_against` probes rvalue binding while call sites pass const lvalues — satisfying-but-then-erroring testers fail in a confusing place (`FreeCheckers.hpp:184-190`); `log_summary`'s unsigned subtraction can underflow for nested performance checks via direct free-function use (`TestLogger.cpp:279-283`); `object_generator` is constrained `movable` but stores in copy-requiring `std::function` (`StateTransitionUtilities.hpp:49-52`).

## Tier 3 — questions where intent decides

- **The `// TO DO: Bug here with #endif`** (`DependencyAnalyzer.cpp:150`): traced `#endif` variants and couldn't construct a failure from `#endif` itself — the live hazards are neighbors: `# include` / `#include"x.hpp"` are never recognized (under-selection), and directive tails aren't consumed, so a cutoff word in a tail terminates include collection early. What did the TODO originally refer to?
- **Allocator-extended copy ctors apply SOCCC to the supplied allocator** (`PartitionedData.hpp:107-108, 906-917`) — standard containers don't; a pmr-style allocator passed explicitly would be silently replaced, with a knock-on interaction in `assignment_helper`'s no-propagation branch. Deliberate?
- **Is `thread_pool::push` meant to be single-producer?** `m_QueueIndex++` and `joined` are unsynchronized; if multi-producer is intended, `atomic fetch_add(relaxed)` is a drop-in — if not, worth documenting.
- **Dead directed-embedded branches in `validate()`** (`Connectivity.hpp:998-1057` plus `check_embedded_edge`/`check_inversion_consistency`/`edge_inversion_info` in GraphErrors): unreachable since `is_embedded` implies `!is_directed`, and they reference members no current edge type has. Groundwork for reinstating directed-embedded graphs, or cleanup fodder?
- **Tree `prune` assumes `add_node`-built index ordering** (`DynamicTree.hpp:106-117`) but `tree_base` exposes `sort_nodes`/`swap_nodes`/positional `insert_node`, which violate it. Defend or restrict?
- **Every versioned `*_FPOutput.txt` entry is headed "False Negative Failure:"** (`TestLogger.cpp:113-136`, via the ironically-named `fpMessageMaker`) — residue of the historical mode rename?
- **`asynchronous::push` uses `async | deferred`** (`ConcurrencyModels.hpp:185`) — a never-`get()`d future may never execute. Truer to the name would be plain `async`; deliberate?
- **`suite` CTAD deduces reference members from lvalues** (`Suite.hpp:51`), and `extract_leaves` then moves from the caller's objects through a by-value suite (`Suite.hpp:216`). Latent (the runner passes rvalues) — intended pilfering or missing value-semantics constraint?
- **`with_best_available` skips `operator==` when a same-type equivalence tester exists** (`FreeCheckers.hpp:636-686`; branch 3 lacks the `binary_comparison` that branch 1 adds) — for `fs::path` that's content-equivalence *instead of* lexical equality; the asymmetry looks unintended.
- **`mem_ordered_tuple`'s layout rests on implementation-defined base ordering** (`MemOrderedTuple.hpp:37`) — fine on all three ABIs, but the one property the type exists for has no `offsetof` static_assert pinning it.
- **Sentinel destructor bookkeeping during stack unwinding** (`TestLogger.cpp:99-145`): an exception escaping a top-level FN-mode check records a mode-failure *in addition to* the runner's critical failure — noise/double-count rather than false pass, but worth deciding deliberately. (Advisor exceptions propagate correctly outside the destructor; allocation failure inside the destructor's message building is terminate — probably an acceptable trade.)
- **`copy_special_files_back` iterates non-recursively** (`MaterialsUpdater.cpp:28-54`) yet computes relative dirs as if recursion were intended — `.seqpat`/`.keep` in prediction subdirectories are never copied back. Oversight?
- **Performance-test statistics** (`PerformanceTestCore.hpp:78-203`): the acceptance band uses sample SD of individual timings (not standard error of the mean, so it doesn't narrow with trials), and the retry loop breaks on first pass — both choices push toward eventual passing. Known instability compromise, but both levers push the same direction; also the doc comment at lines 45-51 states the wrong condition *and* conclusion vs the code at line 147.

## Design weaknesses (recurring themes)

- **Silent tester non-participation is the framework's one genuinely silent failure mode.** Concept-gated dispatch means a mis-signatured `value_tester` overload (wrong tag, non-const ref, extra parameter) quietly doesn't match, and `check` proceeds to the range branch or a fallback — possibly weaker than the author intended, with no diagnostic. The FN/FP machinery cannot see this class: it only exercises testers that *do* participate. A compile-time probe — "a `value_tester<T>` specialization exists but no overload matched" — would close it, and dovetails with the `tests_against` rvalue-probing mismatch.
- **Interchangeable policies with divergent contracts**: bucketed vs contiguous storage disagree on OOB partition access, insert clamping vs throwing, and empty-container `begin_partition(0)`; the three static adaptors disagree on empty-`pop`; within one class, some misuse throws while `erase_slot`/`insert_slot`/`swap_partitions` silently no-op and `insert_slot(pos)` past the end silently appends. Iterator-invalidation rules — the most important contract for containers trading in long-lived iterators — are documented nowhere. `end_partition(i) == begin_partition(i+1)` compares true across partitions (base-iterator-only comparison) — probably intended, deserves a doc sentence.
- **[Roadmap-relevant] The factory will fight the reflection plan as-is**: the all-products-constructible-from-one-argument-list constraint (`Factory.hpp:107-108`), the variant vessel sized to the largest product, and `product_creator` hard-coding `producer<Product, Product>` (`Factory.hpp:80-81`) while the `creator` concept and Handlers machinery — the right vocabulary for producing e.g. `unique_ptr<test_base>` — already exist but aren't wired in. Also: no `contains(name)`/enumeration introspection (the "did you mean" UX will want it); `nomenclator`'s deleted primary members document the wrong interface (`name()` vs `operator()`); `meta::type_name` is compiler-dependent, so it can't feed user-facing string keys without normalization; `suite` forbids mixed suite/leaf children (`Suite.hpp:37`) though `leaf_extractor` already handles heterogeneity; `granular_filter`'s suite-name matching is hard-coded to exact `equal_to` (`Suite.hpp:388`).
- **Single-slot extender composition** (`checker<Mode, Extender>`, `FreeCheckers.hpp:708`): a test wanting regular semantics *and* performance checks has no supported combination; a documented protected `logger()` accessor would decouple this today, ahead of variadic friends.
- **Monoliths in flux-adjacent places**: `Connectivity.hpp` (1,749 lines, six concerns; the comp-index disambiguation heuristic in `modify_comp_indices` (:1571-1580) is correct but undocumented — the full stale-vs-corrected case analysis belongs in a comment); `Spaces.hpp`/`PhysicalValues.hpp` mixing essay, concept tower, and instantiations.
- **Platform-determinism seams**: `priority_first` tie-breaking is unspecified-order (versioned outputs across three platforms deserve a deterministic tie-break, e.g. by node index); true-recursion `depth_first` overflows on ~10⁵-node paths (`GraphTraversalDetails.hpp:559-574`); stateful traversal visitors are copied per-visit in every accumulator except `serial<void>`, contradicting the in-code comment (:158, 198, 364-366).
- **MSVC-debug prediction shifting silently discards user-supplied deltas** (`AllocationCheckers.hpp:185-196` replaces rather than composes) — exactly on the platform the shifting exists for; if the two-arg `alloc_prediction` ctor is internal plumbing, making it private/befriended would express that.
- **Runner odds and ends**: `read_modify_write` no-ops on unreadable files, so `add_include`/`add_to_cmake` silently skip while `add_to_suite` throws — a `create` can half-succeed with no diagnostic; per-build-dir discrimination stops at prune (TestSummaries/DiagnosticsOutput/TestsTemporaryData are shared — two configs running simultaneously race working copies); CLI numeric parsing lets raw `std::stoi` exceptions escape for `--thread-pool`/`--runner-id`; `m_RunnerMode &= runner_mode::help` is "clear all modes" in a bitmask costume; `path_equivalence` does a full recursive directory scan per unmatched candidate per test; the hardcoded pool-size 8 appears twice; `run_tests` dereferences `wt.optTest->parallelizable()` unchecked, relying on the (non-stable) sort partition — an assertion would make the invariant explicit; `regular` `check_semantics` returns `void` while move-only returns `bool`; regular allocation tests hardwire empty moved-from predictions while the move-only API accepts them; `individual_allocation_predictions` derives para predictions invisibly from the copy prediction.
- **Template/instantiation weight**: `add_type_info` → `demangle` runs on every check (hot path); `regular_extender` repeats twelve near-identical forwarding overloads; the 8-way propagation driver multiplies it; the empty/non-empty `NodeWeight` `graph_base` specializations duplicate ~15 constructors and the meta-data `undirected_graph` specialization duplicates the whole class body; `node_comparer` fails with a deep instantiation error rather than a "priority traversal needs node weights" static_assert. None wrong; this is where compile times will hurt first.
- **`STATIC_CHECK` macro hygiene** (`FreeCheckers.hpp:809`): globally scoped from first include, never `#undef`'d, name-collides with Catch2's `STATIC_CHECK`; unqualified `check` presumes expansion inside a checker member; the `[&]` capture is unused. Consider `SEQUOIA_STATIC_CHECK` or documenting the collision.

## Gaps

**Process (the big one).**
- **No CI of any kind.** The 3-compiler × Debug/Release × ASan matrix, the git-diff DiagnosticsOutput oracle, and the end-to-end test all run only when manually invoked. A minimal job — Linux gcc, build TestAll, run, `git diff --exit-code output/` — converts the passive oracle into an active one; Windows/mac legs matter most because that's where versioned outputs drift silently. Even without CI, an opt-in "runner exits nonzero if versioned diagnostics changed" flag would close the loop locally.
- **No TSan preset; the thread pool has never run under a race detector.** Its unit test is 145 lines pushing one task per model — the thinnest coverage relative to risk in the library, for a component whose failure corrupts *test results*. A TSan preset plus a functional stress test (many tasks × threads, every future satisfied exactly once, double-join, push-after-join, zero-thread) is cheap. ASan is the only sanitizer configured; no UBSan preset either.
- Checked-in coverage report (94.0% lines / 91.6% functions) is dated 2025-11-09 — four months stale; TestFramework itself is the lowest-covered directory at 88.1% lines, and the uncovered part is by definition the part with no self-diagnostic.
- Build system: no `/WX`/`-Werror` anywhere (warnings can accumulate unnoticed without CI); the `cxx_std_23`-on-Windows / `cxx_std_26`-elsewhere split deserves at least a comment; blanket clang/gcc `-Wno-*` suppressions with nothing recording their cause; `base-asan-msvc` overwrites `CMAKE_CXX_FLAGS` wholesale (footgun if base flags are ever added); `ctest` on a normal configure silently runs zero tests (tests only registered under `CODE_COVERAGE`).
- Blind spots inherent to the FP/FN + versioned-output scheme (worth knowing, not necessarily fixing): the DiagnosticsOutput oracle is passive (detection = human reads `git status`); FN mode proves specific enumerated bad inputs are flagged, not input classes; `.seqpat` masking erases exactly the regions where timing/path/demangler bugs would manifest; in-process runner tests pin `--serial`, so concurrent-runner correctness rests on end-to-end output equivalence plus daily use; the framework's tests run inside the framework (mitigated genuinely by FN inversion and nested-runner stream capture, but the outermost shared `check()` is a single point of failure).

**Checking vocabulary.** No `failure_reporter<std::ranges::not_equal_to>` ("check these differ" degrades to a bare bool with no value reporting); no relative-tolerance/ULP comparator (`within_tolerance` is absolute-only, unconditionally `noexcept` around user ops — a throwing user `abs` terminates instead of becoming a critical failure — and can UB on integer overflow); **`check_semantics` never checks self-move-assignment** — the classic third member of the self-copy-assign/self-swap family it already covers; `y`'s copy constructor is never directly verified (only `x`'s — a copy ctor correct for `x`'s state but not `y`'s shows up only indirectly); `null_allocation_event::comparison` is never referenced anywhere — deliberate vocabulary or leftover?

**Graph algorithms.** Exactly one algorithm (`sub_graph`: weights-only predicate, no edge/index-predicate variants, O(V·E) via repeated `erase_node` on a full copy). Customers — including sequoia's own DependencyAnalyzer — would expect: connected components; topological sort + cycle detection (the analyzer's DFS post-order propagation quietly under-propagates if an include cycle ever appears — a cycle check makes that loud); shortest paths (the priority-first scaffolding is 90% of Dijkstra); an isomorphism/canonical-form answer to "same graph, different labelling" (`operator==` is representation equality, documented but limiting). Each is a thin composition over `traverse` — lecture-sized.

**Sequences/Statistics.** `linear_sequence` has `operator[]` but no `begin`/`end`/`size` — can't range-for or feed ranges algorithms; `monotonic_sequence`'s `unsafe_t` contract ("caller restores monotonicity") is documented nowhere — its only production caller preserves it, but nothing enforces that discipline; no median/quantiles, no covariance, no single-pass (Welford) variance — the last matters because the natural framework use (performance instrumentation) wants streaming; the default `sample_standard_deviation` using the Gaussian n−1.5 correction is defensible but unusual and deserves a doc comment.

# Addendum (2026-07-17): graph-based testing — `StateTransitionUtilities.hpp`

**Scope:** `transition_checker`, `transition_info`, `object_generator`, plus the traversal they sit on (`traverse(breadth_first, g, find_disconnected_t{0}, ...)`) and both customer populations (sequoia's graph/diagnostics tests; avocet's `CapabilityManagerFreeTest`). Findings traced by reading, not compiled repros. Prior coverage: item 23 already records the `object_generator` movable/`std::function` mismatch (:49-52).

## Strengths

- **Full-coverage traversal by construction**: `find_disconnected_t{0}` restarts BFS at the lowest undiscovered node, so every node is processed and every edge fires exactly once even for disconnected transition graphs — a test *cannot* silently skip a subgraph. The opposite failure direction from the prune findings, and worth preserving through any rename/refresh.
- **Ordering is compulsory data, not optional decoration**: when `T` is `totally_ordered`, `CheckOrdering` defaults to `yes`, the edge weight grows a `std::weak_ordering` member, and — since `weak_ordering` has no default constructor — aggregate init *fails to compile* if an edge omits it. Opt-out is explicit (`check_ordering::no`). This is the "make the safe thing the default, make the exception loud" pattern done right.
- **Progressive disclosure via the CheckFn overload family**: (desc, obtained, prediction) → +parent → +node indices → +ordering → fully lazy `std::function<T()>` triple. The lazy form simultaneously solves move-only support, evaluation-order control, and `check_semantics` reuse.
- **Self-hosting**: the maths graph is the data structure of the test facility, and the graph tests are themselves `transition_checker` customers.
- **The avocet capability test demonstrates the intended bridge for context-keyed state**: `T` is a pure payload value; each transition first forces the GL context to the parent state (`ctx.new_payload(hostPayload)`) before applying the target — making every edge self-contained and traversal-order-independent despite the hidden mutable context.

## Findings

- **A1. Unspecified evaluation order between transition and prediction** (`StateTransitionUtilities.hpp:91-94`): in the eager `invoke_check_fn`, `w.fn(parentGenerator())` (the transition) and `g.cbegin_node_weights()[target]()` (the prediction) are indeterminately sequenced function arguments. Benign for value-semantic `T` with pure generators — all current customers — but the move-only story explicitly invites *generator lambdas*, and a generator that touches shared state (e.g. constructing a GL resource against the live context) would make obtained/prediction order-dependent, i.e. potentially different between MSVC and gcc. Hardening is two named locals evaluated in a fixed order. [PLAUSIBLE-consequence, CONFIRMED-mechanism]
- **A2. A throwing transition loses its edge attribution** (whole `check` family): the carefully built "Transition from node i to j" message is passed to CheckFn, but the transition itself runs during argument evaluation — an exception (e.g. avocet's GL-error throws) propagates past the message to the test-level handler, and the remaining edges go unchecked. Catch-per-edge (append the edge message, rethrow or log-and-continue) would match the framework's failure-attribution philosophy. Real for avocet today: any capability transition that trips a GL error reports without saying which edge.
- **A3. Ordering overloads aren't constrained on `CheckOrdering`** (:150-162, :174-183): they require `std::totally_ordered<T>` but not `CheckOrdering == check_ordering::yes`. For `transition_checker<T, check_ordering::no>` with ordered `T`, an ordering-taking CheckFn selects the overload and then hard-errors on `i->weight().ordering` (the primary `transition_info` has no such member) — the constraint admits what the body can't do.
- **A4. `object_generator` init-check ctor: constraint checks `std::string`, call passes `std::string_view`** (:62-66): `std::invocable<InitCheckFn, std::string, T, Args...>` admits an InitCheckFn taking `std::string` by value, but the body calls `initCheckFn(message, ...)` with a `string_view` — and string_view→string is an *explicit* conversion, so exactly the functor the constraint blesses fails to compile. Same family (value-category edition): the public `check` overloads constrain on rvalue `std::string` but invoke with a const lvalue, so a `std::string&&` CheckFn is admitted-then-rejected. Align constraints with call sites (`const std::string&` / `std::string_view`).
- **A5. Extension of item 23**: because the `Args...` convenience ctor (:54-59) and the init-check ctor (:61-67) both delegate to `object_generator(T)`, the *entire* convenience-constructor family is unusable for move-only `T`, not just the stored-value ctor — in particular move-only + init-check is currently unexpressible; move-only users must hand-write generator lambdas (as `MoveOnlyStateTransitionDiagnostics` does).
- **A6. In-degree-0 nodes are silently unverified**: a node's weight is only ever *checked* when some edge targets it; a root (or isolated) node's claimed state is trusted as transition input. The init-check ctor is the designed answer, but nothing flags a node that has neither in-edges nor an init-check — a cheap post-traversal diagnostic would close it, in the same spirit as the tester-participation diagnostic.
- **A7. Implicit contract worth documenting**: generators are invoked multiple times per edge (the parent-passing overloads call `parentGenerator()` at the call site *and* inside `invoke_check_fn`), so generators must be idempotent and cheap-ish; with impure generators this also compounds A1.

**Nits**: redundant `public:` as first section of `struct transition_checker` (:76-78); missing space in `std::totally_ordered<T>&& pseudoregular<T>` (:151 — parses fine, reads as an rvalue ref); the lazy path wraps an `object_generator` (itself a `std::function`) in another `std::function` — double type-erasure per edge (:104); `invoke_check_fn` copies the parent `object_generator` and `Args...` by value per edge (test-time cost only); stray double blank line (:184-185).

## Avocet-side observations (`CapabilityManagerFreeTest.cpp`)

- Every edge description is `report("")`, so a runtime failure reads "Transition from node 9 to 13" — the `node_name` enum aids the *author* but not the failure *reader*. Either populate descriptions ("stencil_test → configured_stencil_test") or — nicer, sequoia-side — let nodes carry names that `make()` splices into the message instead of bare indices.
- The graph is a hand-sampled, asymmetric subset of the 14×14 transition matrix (blend gets a self-edge and a cross-edge to multi_sample; depth_test and stencil_test get no cross-edges; sample_alpha_to_coverage gets a self-edge, sample_coverage doesn't). Deliberate spot-sampling, or accretion? If the capability manager treats capabilities independently, sampling is defensible — but the full matrix is a ~20-line nested loop over the enum and the 240-line literal initializer collapses with it (unless the literal graph is the lecture artifact, in which case leave it).

## For the planned rename

`graph-based-testing` describes the *technique*; the header's contents (`transition_checker` + supporting types) are one facility implementing it. If the refresh grows node-naming (above) or per-edge exception capture (A2), the rename is the natural moment. A1/A3/A4 are each small, contained fixes that could ride along.

# Addendum (2026-07-20, part II): critical review of the test-materials framework

**Method:** three parallel agents — (i) principles & client experience, (ii) architecture, (iii) line-level code scrutiny with compiled verification probes (MSVC 14.51; findings marked CONFIRMED were traced or probe-verified). Paths relative to `dependencies/sequoia/Source/sequoia/` unless noted.

## A. Correctness — today-bugs

- **T1. The whole update pass runs inside a `noexcept` function — any failure is `std::terminate`** (`TestFramework/TestRunner.cpp:189-200`). `test_tracker::decrement_depth() noexcept` runs every `soft_update` plus `update_prune_info()`. `soft_update` throws freely: `std::regex_error` from one malformed `.seqpat` line (probe-confirmed), `filesystem_error` from read-only/locked prediction files, T2's type-change throw, the `logic_error` symlink tripwire. Trigger: `u` run + one bad regex line → terminate *after all tests ran*, summary/prune state lost, prediction tree potentially half-merged. In the check path the same `regex_error` is caught and logged; only the update path detonates. Fix independent of everything else: make the update pass an explicit, exception-handled phase (per-updateable try/catch, report, continue). CONFIRMED.
- **T2. Same-name file→directory change makes `soft_update` throw; direction asymmetric** (`TestFramework/MaterialsUpdater.cpp:128-141`). `compare` orders regular < directory, so the additions loop copies the working *file* while the prediction *directory* of the same name still exists — `fs::copy` throws (probe-confirmed; Windows message: "Access is denied"). Reverse shape (dir→file) works because removal precedes copy. Lands inside T1's `noexcept`. Neither shape is in the `MaterialsUpdaterFreeTest` fixtures. Fix: remove same-relative-path/different-type prediction entries before the copy loop. CONFIRMED.
- **T3. `read_to_string` opens in text mode** (`Streaming/Streaming.cpp:47`). Probe-confirmed: on MSVC a file containing 0x1A truncates at the Ctrl-Z. Binary materials (images, blobs) through `string_based_file_comparer` can **falsely pass**, and `soft_update`'s reduced-content gate sees truncated-equal → **never updates the genuinely changed file**. Separately, LF-vs-CRLF content compares equal on Windows, unequal on POSIX (distinct from known M3). Fix: `std::ios::binary` (+ explicit newline normalization if desired). CONFIRMED.
- **T4. Degenerate paths in `value_tester<path>::check_path`** (`TestFramework/ConcreteTypeCheckers.hpp:463-493`). Empty vs empty/nonexistent: `not_found == not_found` passes, `if(!path.empty())` skips the rest → **silent green having verified nothing** — reachable, since the accessors return `""` for missing materials/Prediction. Non-empty-nonexistent vs empty: `back(prediction)` throws "Cannot extract final element..." — misleading. Both nonexistent: "not currently implemented" `logic_error`. Fix: explicit `not_found` rejection + a deliberate decision about empty. CONFIRMED mechanics.
- **T5. The two tree-walkers disagree** (`value_tester<path>` vs `soft_update`). (a) *Exclusions*: tester excludes `.DS_Store`/`.keep`/`*.seqpat` (`ConcreteTypeCheckers.hpp:449-453`); `sort_dir_entries` (`MaterialsUpdater.cpp:75-85`) excludes nothing and staging copies unfiltered — a `.DS_Store` in working gets **committed into the version-controlled Prediction tree where no test can ever flag it**; prediction-side files the test never compared can get deleted. (b) *Weak equivalence*: tester pairs positionally ignoring the root name; updater pairs strictly by name — a weak-passing `a.txt`/`b.txt` pair gets churned (add `a`, delete `b`) on update. (c) *Type mismatch*: tester reports cleanly; updater throws (T2). These are one reconciliation algorithm rendered twice; the architectural fix is a single tree-walk engine emitting events (`extra_in_working`, `missing`, `type_mismatch`, `content_differs`) with exclusion + equality policy injected, consumed by both the tester (→ logger checks) and updater (→ copy/remove).
- **T6. Comparison policy is per-call-site; update policy is global** (`ConcreteTypeCheckers.hpp:387-391` vs `MaterialsUpdater.cpp:108-116`). Bespoke per-extension comparers are invisible to the updater, which hard-codes reduced-string equality. Laxer comparer (tolerances): update overwrites every byte-differing file → run-varying noise ratcheted into VCS. Stricter comparer: reduced-equal but comparer-rejected file is never overwritten → **`update-materials` never converges**. Also: the updater applies seqpat reduction even to files whose comparison never did. Unwritten contract today: "custom comparers void the update-materials warranty". Fix direction: comparer policy discoverable by the runner (trait/virtual on the test, or a manifest in `Prediction/`), consumed by the T5 engine.
- **T7. Shared staged WorkingCopy × parallel-by-default = unguarded race** (`TestRunner.cpp:281`, `FreeTestCore.hpp:249-250`). Tests sharing a source file share one staged working dir; all concrete tests default parallelizable and run under `par`. Two same-file tests that *write* race on the shared dir (sequoia's own same-file pairs only read — never tripped). The dedup hit at `set_materials` is the exact detection point: force such tests into the existing serial prefix, or discriminate staging dirs. (Verified non-race: `process_test`/`m_Updateables` run in the serial post-traverse — the tracker itself is safe.) **Disposition (user, 2026-07-20): subsumed by a planned rework — the materials directory will gain the test class name as a subdirectory, making materials truly test-specific rather than file-specific (likely riding the reflection-based registration improvements, though not strictly requiring them); `DiagnosticsOutput` naming gets the same treatment since two classes in one source file can in principle clash there too. Do not propose serialization workarounds as the long-term fix.**
- **T8. Mode is a call-time filesystem query** (`IndividualTestPaths.cpp:107-141`). Every `working()`/`prediction()`/`auxiliary()` re-stats the original tree: (a) staging-time vs test-time answers can differ (a mid-run `git checkout` suffices); (b) `operator==` claims value semantics the class doesn't have; (c) `prediction()`/`auxiliary()`/`original_auxiliary()` lack the `m_Materials.empty()` guard that `working()` has — on a default-constructed instance they evaluate `fs::exists("Prediction")` **relative to the CWD**, so a stray `Prediction` dir in the CWD yields a non-empty bogus path. Also platform-divergent: `prediction/` (lowercase) works on Windows/macOS, silently flips to flat mode on Linux. Fix: resolve mode once at construction (enum `absent`/`flat`/`trichotomous`), `optional<path>` over `""`-sentinels, near-miss validation on the three magic names.
- **T9. `fs::relative` where lexical is meant resolves symlink leaves** (`MaterialsUpdater.cpp:80`, `ConcreteTypeCheckers.hpp:530`). Per [fs.op.relative] the leaf symlink canonicalizes — a `dir/link → ../elsewhere` entry yields `relative == "../elsewhere"`, so the merge can `create_directory`/`copy` **outside the prediction tree**, and the tester mis-reports pairing. ## B. Client experience & design

- **T11. seqpat semantics under-deliver**: (a) *deletion-to-empty* erases presence/count information — working "took [4ms], done" matches a prediction that never had a timing at all; a fixed placeholder token (`<scrubbed>`) preserves the intended equality classes, costs one literal, and keeps update gating consistent automatically (approvaltests-style scrubbers get this right). (b) Patterns run against the whole file as one string with default ECMAScript — **no `multiline`**, so grep-style `^...$` patterns are silently dead; dead patterns are observationally invisible until noise leaks. (c) Failure coordinates ("first difference on line N") index the *reduced* text that exists nowhere on disk, and nothing discloses that reduction happened — append "after reduction via Foo.seqpat" to the preamble. (d) One `Foo.seqpat` governs *all* same-stem files (`Foo.txt` and `Foo.json`) — invisible sharing; append-style naming (`Foo.txt.seqpat`) would fix at the cost of a materials migration.

## E. Strengths worth preserving

- **Staging into `output/TestsTemporaryData`** — disposable mutable copies, originals untouchable by construction, no `*.received.*` litter (avocet's flat-mode use shows it earning its keep for pure inputs too).
- **Reduced-content-gated updates are ahead of the mainstream** — insta/Jest rewrite snapshots wholesale on accept; sequoia only overwrites files whose *reduced* content differs, and "git diff is the review surface" makes per-file accept fall out of the VCS for free.
- **Colocated, version-controlled, declarative scrubbers** participating in both comparison and update gating through one choke point — better factoring than code-side scrubbers that don't gate updates.
- **The `copy_special_files_back` transplant trick** — normalize the input so one uniform merge does the right thing, re-applied per level by the recursion.
- **Deferred, deduplicated updates** — all verdicts computed against pristine predictions; keep this sequencing when fixing T1.

**Priority view:** T1+T2 first (the flagship `u` workflow can terminate the process on inputs as mundane as one bad regex line or a file→dir output change, leaving a half-merged prediction tree); T3 next (false passes on binary materials); then the T5/T6/T14 cluster, which is one refactor — a single reconciliation engine with injected policy — rather than three fixes. T11a (placeholder over deletion) and T12 (documentation) are the highest-value/lowest-cost client-experience wins.

# Addendum (2026-07-21): `check_exception_thrown` & exception-message postprocessing

**Scope:** `check_exception_thrown` (`FreeCheckers.hpp:324, :804`), `default_exception_message_postprocessor` (`FreeCheckers.hpp:299-313`), `exception_message_extractor`, and the versioned-exceptions pipeline (`log_caught_exception_message` → `versioned_write` → `output/DiagnosticsOutput/**/*_Exceptions*.txt`). Context: the postprocessor exists so version-controlled exception files stay diff-stable across platforms/checkouts. Avocet supplies **no custom postprocessor anywhere** — stability comes from generic-format path discipline at throw sites (`Formatting.cpp:17`, `ShaderProgram.cpp:137,143`, `Image.cpp:33,46`) + the default root-strip + curlew's `output_discriminator` platform suffixing for genuinely driver-dependent text.

## Findings — for Oliver to look at

- **E1. First-occurrence-only root stripping** (`FreeCheckers.hpp:305-309`): the default postprocessor erases only the first occurrence of `project_root().generic_string()`. A message carrying two absolute paths ("failed to copy X to Y") keeps the root in the second — quietly re-introducing checkout-location diffs in versioned output. No current message has two paths; latent. **Already on Oliver's own TODO list (confirmed 2026-07-21).**
- **E2. The generic-format contract is invisible**: the default only matches the *forward-slash* form of the root. A message built from a native-format path — `fs::filesystem_error::what()` (native separators), or raw `source_location::file_name()` unlaundered — won't match on Windows, leaking an absolute backslashed path into version-controlled output on one platform only. Avocet's throw-site discipline upholds the contract, but nothing documents or enforces it. Options: also try the preferred-separator form in the default postprocessor, and/or state the contract in the docs.
- **E3. No garbage collection of versioned output for deleted tests**: sequoia overwrites versioned files for tests that run, but nothing removes files for tests that no longer exist — deleting a test leaves its `TestSummaries`/`DiagnosticsOutput` files committed forever. Found via avocet: `InfiniteLoopFreeTest` (deleted 2025-09-04) left 12 summaries + 1 exceptions file; `VertexBufferObject*` (superseded by the generalized buffer-object tests, `a52a9f13`) left 2 summaries + 1 FN output. All 15 manually removed 2026-07-21, orphan sweep confirmed clean. **Automatic GC is already on Oliver's own TODO list (confirmed 2026-07-21).**
- **E4. Text-mode writes of versioned output cause recurring phantom line-ending diffs on Windows** (write-side twin of T3): every writer of committed files opens text-mode — `write_to_file` (`Streaming/Streaming.cpp:59`, behind `versioned_write`), the summary writer (`TestRunner.cpp:251`), the prune writers (`DependencyAnalyzer.cpp:109,541`). Windows runs rewrite committed output CRLF against the repo's `* text=auto eol=lf` policy; any file whose working copy is freshly checked out LF (pull/branch-switch/commit touching it) flags as modified with identical content, and periodic renormalization only suppresses it until the next refresh. Oliver hits this recurrently (2026-07-21). Fix: `std::ios_base::binary` on these streams (LF everywhere, bytes identical to checkout — the phantom class becomes structurally impossible); natural to land together with T3's binary reads, making the versioned-output pipeline byte-symmetric. No good repo-side alternative (un-texting `output/**` would commit CRLF → real cross-platform diffs).
- **E5. Nits**: (a) `pos < npos` (`FreeCheckers.hpp:305`) is a quirky spelling of `!= npos`; (b) `erase(pos, len + 1)` assumes a separator follows the root — a message ending in exactly the root eats one extra char; (c) the IIFE at `FreeCheckers.hpp:326-330` (`std::move(append_lines(description, ...))` returned as `std::string&&`) is correct — the lvalue `append_lines` overload builds into `description` in place — but reads as dangling until the overload set is checked; mutate-then-move would say the same with less suspense.
