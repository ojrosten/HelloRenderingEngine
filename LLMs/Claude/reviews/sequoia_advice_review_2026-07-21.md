# Sequoia: the Advice framework — review

**Date:** 2026-07-21
**Scope:** `Source/sequoia/TestFramework/Advice.hpp/.cpp`; the advice plumbing in `FreeCheckers.hpp`, `ConcreteTypeCheckers.hpp` (string tester's `make_advisor`), `SemanticsCheckersDetails.hpp`, `AllocationCheckersDetails.hpp/.cpp` (`allocation_advice`); tests in `Tests/TestFramework/ElementaryFreeDiagnostics.cpp` + `ElementaryFreeDiagnosticsUtilities.hpp` (`bland`); versioned outputs consulted as evidence.
**Method:** single-session direct reading (no agents), claims verified against source, `git log -S` history, and the laid-down FN outputs. Prior-review overlaps attributed, not re-raised: the `std::move` of a `const tutor&` (ConcreteTypeCheckers.hpp:277/283) and the missing space at FreeCheckers.hpp:680 are already in the 2026-07-16 review's nit set.
**Register:** settled code (framework core, in production use); findings are graded defect/question/nit accordingly.

**Verdict up front.** The design is strong and the implementation largely faithful to it: advice is opt-in per check, value-dependent, and propagates through `value_tester` decomposition until the advisor's `operator()` can bind — with mismatched advisors *silently ignored* rather than erroring, and implicit-conversion binding deliberately permitted (widening) while single-overload narrowing is guarded against. Highlights: the `advisor_analyser` dispatch (Advice.hpp:60-89); the string tester's `make_advisor` nesting char-level advice inside the substring context (ConcreteTypeCheckers.hpp:97-121); internal dogfooding via `allocation_advice` for the counter-intuitive negative-count case; and diagnostics tests that cover utilized/ignored/conversion-bound/multi-overload (`bland`) advice plus the fallback chains. The findings below are edge-of-envelope: dead metafunctions, one test that doesn't test what its name says, two constraint-boundary gaps that turn graceful-ignore into hard errors, and one hole in the narrowing guard.

---

## Substantive (AD1–AD5)

- **AD1 [verified via git] Dead code: `is_teacher_v`, `ends_with_tutor`, `ends_with_tutor_v`** (Advice.hpp:161-178). Their last consumer (`equivalent_type_processor` in FreeCheckers.hpp) was removed 2024-12-30 by 826d4685d "Simplify the core testing framework functionality"; grep of `Source/` and `Tests/` finds no remaining customers (generated doxygen HTML still shows the old use — regenerates away). Removal candidates. If kept deliberately, the `std::get`-on-a-tuple-of-references trick for last-of-pack could become `std::tuple_element_t`.

- **AD2 [verified against versioned output] Two "Float check" advice tests actually operate on doubles** (ElementaryFreeDiagnostics.cpp:104, 107 — literals `6.5, 5.6`; `ElementaryFreeDiagnostics_FNOutput.txt` shows `[double]` for both). Line 104 ("advice for ints ignored") duplicates the double variant at line 89; line 107 ("advice for doubles utilized") is trivially double-advice-on-doubles, so the case it appears to intend — *widening* binding of a double advisor onto **float** values via the `common_type` rule — is untested. Presumably `6.5f, 5.6f` intended (cf. line 101's genuine float literals). Fix churns the versioned output (`[double]` → `[float]`, value formatting).

- **AD3 [question — is rvalue-invocability deliberate?] Constraints test rvalue invocation; the calls pass const lvalues.** `advisor_analyser`'s generic case requires `std::invocable<Advisor, T, T>` (Advice.hpp:67) and `advice_data`'s constrained ctor requires `is_invocable_r_v<std::string, tutor<Advisor>, T, T>` (:189), but `tutor::operator()` and the `advice_data` body invoke with `const T&` lvalues (:139-141, :192). Consequences: an advisor with `operator()(T&&, T&&)`, or by-value parameters where `T` is move-only (e.g. advising on a `unique_ptr` check), satisfies the constraint and then hard-errors inside `tutor::operator()` instead of being gracefully ignored. `std::invocable<Advisor, const T&, const T&>` matches the call as made, converts both hard errors into the documented silent-ignore, and leaves every currently-working advisor unaffected. Same shape at the string tester's `make_advisor` (`invocable<tutor<Advisor>, Char, Char>`, ConcreteTypeCheckers.hpp:100) — harmless for scalar `Char`, but consistency argues for the same spelling.

- **AD4 [question — acknowledged-partial guard, but this hole misleads] The narrowing guard admits integral→floating binding.** The single-overload rule `common_type_t<param, T> == param` (Advice.hpp:85) lets a `(float, float)` advisor bind to a `long long` check, since `common_type<long long, float>` is `float` — a narrowing conversion per [dcl.init.list], and genuinely lossy: for obtained/predicted `16777217LL` vs `16777216LL` the advisor receives `16777216.f` *twice* — equal values on a failing check, so value-dependent advice is actively misled. This differs in kind from the permitted widenings (which preserve value): the comment says "Attempt to disallow", so partial coverage is acknowledged, but a supplementary rejection for `integral<T> && floating_point<param>` where the mantissa can't hold the integer's digits (`numeric_limits` digits comparison) would close the information-losing direction.

- **AD5 `advisor_invoke_type` hard-errors on C++23 `static operator()`.** A single-overload static lambda — `tutor{[](int, int) static { return "…"; }}` — passes the `&Advisor::operator()` requires-check (Advice.hpp:77-79), but the resulting plain function-pointer type matches no `advisor_invoke_type` specialization, so the undefined primary gives an incomplete-type hard error inside `advisor_analyser` (:83). Two more specializations (`R(*)(T,T)` + `noexcept` variant) cover it. Ref-qualified `operator()` has the same shape but is exotic enough to ignore.

## Question (AD6)

- **AD6 Minimal-reporting bool funnel forwards the advisor to values it wasn't written for.** In the `minimal_reporting_permitted::yes` branch (FreeCheckers.hpp:678-681) the advisor is forwarded into `check(description, logger, obtained == prediction, advisor)`. Any advisor with bool-convertible parameters — `(int, int)` qualifies, and passes the analyser since `common_type<int, bool>` is `int` — then fires with `(false, true)` ≡ `(0, 1)`: values unrelated to the `T`/`U` the advisor was written against. Since the original values are unavailable at that point by construction, should this branch drop the advisor (pass `{}`) rather than forward it? (Reachable only via the checker's `with_best_available_check_t<yes>` surface; the semantics machinery uses `null_advisor` here.)

## Nits (AD7–AD11)

- **AD7 "propagting"** in the allocation advice string (AllocationCheckersDetails.cpp:20) — the same sentence spells "propagating" correctly four words later. Baked into versioned outputs (`RegularAllocationTestFalseNegativeDiagnosticsBrokenValueSemantics_FNOutput.txt:140,228` at minimum) — regenerate on fix. TestFrameworkPhilosophy.dox:471 quotes this string with the spelling silently corrected.
- **AD8 Doc drift** (documentation-phase input): the philosophy doc's sample output shows a padded, aligned `Advice   :` column (dox:469-472) whereas actual output is `Advice: ` (the tutor-ctor prefix) on its own line; "Clients may provided" (dox:474) → "may provide"; "In the case the `x != 41`" (Advice.hpp:107) → "that".
- **AD9 `tutor` plumbed by value through every internal layer**; the element-wise range loop (FreeCheckers.hpp:403-408) copies it — advisor state plus prefix `std::string` — once per element. Public API by value is fine; `const&` internally drops the O(N) string copies. Same family as the 2026-07-16 `add_type_info` copy nit; distinct site.
- **AD10 `advice_data`'s constrained ctor** assigns in the body rather than the member-init list, and the body is over-indented by two spaces (Advice.hpp:190-194).
- **AD11 `tutor`'s ctor is implicitly converting** (Advice.hpp:131); `Advisor` is never deducible at call sites, so nothing can use the implicit conversion — `explicit` is free.

## Not raised, considered

- Advisors returning non-string-convertible types hard-error inside `tutor::operator()` rather than SFINAE-ing out — but constraining on `invocable_r` would flip the failure to a *silent ignore*, and loud is arguably right for "you wrote an advisor whose return type is wrong". Left as-is.
- `common_type`-undefined pathologies (mutually convertible advisor-param/value types) hard-error in the analyser's `utilize` initializer — real but requires a type pair that converts both ways; too speculative to action.
- The bool `check` overload's advice-on-`(obtained, true)` (FreeCheckers.hpp:688-692) is *by design* and test-documented ("Boolean check with advice").
